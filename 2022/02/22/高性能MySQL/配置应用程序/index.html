第8章 优化服务器设置

从理解MySQL内核和行为开始，然后可以利用这些知识来指导配置MySQL。
最好是从查询语句和响应时间入手来开始分析问题，而不是通过配置项。
另一个节省时间和避免麻烦的好办法是使用默认配置，除非是明确地知道默认值会有问题。

8.1 MySQL 配置的工作原理

MySQL从哪里获得配置信息:命令行参数和配置文件
$which mysqld
$/usr/sbin/mysq1d –verbose –help| grep -A 1’ Default options’
配置文件通常分成多个部分，每个部分的开头是一个用方括号括起来的分段名称。MySQL程序通常读取跟它同名的分段部分，许多客户端程序还会读取client部分，这是一个存放公用设置的地方。服务器通常读取mysqld这一段。一定要确认配置项放在了文件正确的分段中，否则配置是不会生效的。
SHOW GLOBAL VARIABLES
有个特殊的值可以通过SET命令赋值给变量:DEFAULT。把这个值赋给会话级变量可以把变量改为使用全局值，把它赋值给全局变量可以设置这个变量为编译内置的默认值(不是在配置文件中指定的值)。
key_buffer_size：可以一次性为键缓冲区(key buffer,也叫键缓存key cache)分配所有指定的空间。table cache size； thread_cache_size；query_cache_size；read_buffer_size；read_rnd_buffer_size；sort_buffer_size

不要根据一些“比率”来调优:命中率取决于工作负载-某些工作负载就是无法缓存的，不管缓存有多大;其次,缓存命中没有什么意义
不要使用调优脚本
MySQL不是一个完全严格控制内存分配的数据库服务器。

8.3 创建MySQL配置文件

配置服务器的首选途径:了解它内部做了什么，以及参数之间如何相互影响，然后再决定。
有时可以使用SHOW GLOBAL STATUS 的输出，作为配置的输入，以更好地通过工作负载来自定义配置。
每隔60秒来查看状态变量的增量变化:$ mysqladmin extended-status -ri60

8.4 配置内存使用

可以认为MySQL的内存消耗分为两类:可以控制的内存和不可以控制的内存。
在任何给定的操作系统上，MySQL都有允许使用的内存上限。机器上安装了多少物理内存 + 考虑操作系统或架构的限制
在MySQL手册中记载了每个变量的最大值，有需要可以查阅。
MySQL保持一个连接(线程)只需要少量的内存。它还要求一个基本量的内存来执行任何给定查询。
如果没有虚拟内存正在交换(Paging)到磁盘，就是表明操作系统内存足够的最佳迹象。
InnoDB缓冲池并不仅仅缓存索引:它还会缓存行的数据、自适应哈希索引、插入缓冲(Insert Buffer)、锁，以及其他内部数据结构。
MyISAM的键缓存也被称为键缓冲，默认只有一个键缓存，但也可以创建多个。
查询INFORMATION_SCHEMA表的INDEX_LENGTH字段，把它们的值相加，就可以得到索引存储占用的空间:
SELECT SUM(INDEX_LENGTH) FROM INFORMATION_SCHEMA.TABLES WHERE ENGINE=’ MYISAM’ ;
CACHE INDEX t1, t2 IN key_buffer_1;
可把表的索引预载入到缓存中，通过init_file设置或者LOAD INDEX 命令:
L0AD INDEX INTO CACHE t1, t2;
可以通过SHOW STATUS和SHOW VARIABLES命令的信息来监控键缓冲的使用情况。
100 - ( (Key_blocks_unused * key_cache_block_size) * 100 / key_buffer_size)
每秒缓存未命中的次数: Key_reads / Uptime
$ mysqladmin extended-status -r -i 10| grep Key_reads
MySQL键缓存块大小(Key Block Size)
块大小也是很重要的‘(尤其是写密集型负载)，因为它影响了MyISAM、操作系统缓存，以及文件系统之间的交互。
可以设置MyISAM的索引块大小跟操作系统一样，以避免写时读取。myisam_block_size KEY_BLOCK_SIZE
线程缓存保存那些当前没有与连接关联但是准备为后面新的连接服务的线程。thread_cache_size Threads created
表缓存和线程缓存的概念是相似的，但存储的对象代表的是表。每个在缓存中的对象包含相关表frm文件的解析结果，加上一些其他数据。准确地说，在对象里的其他数据的内容依赖于表的存储引擎。
表缓存的设计是服务器和存储引擎之间分离不彻底的产物，属于历史问题。
表缓存分离成两部分:一个是打开表的缓存，一个是表定义缓存(通过table_open_cache和table_definition_cache变量来配置)。
InnoDB有自己的表缓存，可以称为表定义缓存或者数据字典。
可以在MySQL5.6中打开innodb_analyze_is_persistent选项来持久化存储统计信息到磁盘。

8.5 配置MySQL的I/O行为

InnoDB I/O 配置
InnoDB不仅允许控制怎么恢复，还允许控制怎么打开和刷新数据(文件), 这会对恢复和整体性能产生巨大的影响。
InnoDB的缓存和文件
InnoDB事务日志
InnoDB使用日志来减少提交事务时的开销。
整体的日志文件大小受控于innodb_log_file_size和innodb_log_files_in_group两个参数，这对写性能非常重要。
要确定理想的日志文件大小，必须权衡正常数据变更的开销和崩溃恢复需要的时间。
数据大小和访问模式也将影响恢复时间。
当InnoDB变更任何数据时，会写一条变更记录到内存日志缓冲区。在缓冲满的时候、事务提交的时候，或者每一秒钟，InnoDB都会刷写缓冲区的内容到磁盘日志文件。变量innodb_1og_buffer_size可以控制日志缓冲区的大小。
可以通过检查SHOW INNODB STATUS的输出中LOG部分来监控InnoDB的日志和日志缓冲
区的I/O性能，通过观察Innodb_os_log_written状态变量来查看InnoDB对日志文件
写出了多少数据。
“把日志缓冲写到日志文件”和“把日志刷新到持久化存储”不同
高性能事务处理需要的最佳配置是把innodb_flush_log_at_trx_commit设置为1且把日志文件放到一个有电池保护的写缓存的RAID卷中。这兼顾了安全和速度。
使用innodb_flush_method选项可以配置InnoDB如何跟文件系统相互作用。
fdatasync/fsync(); 0_DIRECT; ALL 0 DIRECT; 0_ DSYNC; async_unbuffered; unbuffered; normal; Nosync和littlesync
InnoDB把数据保存在表空间内，本质上是一个由一个或多个磁盘文件组成的虚拟文件系统。
InnoDB用表空间实现很多功能,并不只是存储表和索引。它还保存了回滚日志(旧版本行)、插入缓冲(Insert Buffer)、双写缓冲(DoublewriteBuffer)，以及其他内部数据结构。
通过innodb_data_file_path配置项可以定制表空间文件。
… ibdata3:1G: autoextend:max:2G
SHOW INNODB STATUS的数据可以帮助定位问题。
为了控制写入速度，可以设置innodb_max_purge_lag变量为一个大于0的值。
InnoDB用双写缓冲来避免页没写完整所导致的数据损坏。
SHOW VARIABLES LIKE ‘ myisam_recover_options ‘ ;

8.6 配置MySQL并发

InnoDB有自己的“线程调度器”控制线程怎么进入内核访问数据，以及它们在内核中一次可以做哪些事。最基本的限制并发的方式是使用innodb_thread_concurrency变量，它会限制一次性可以有多少线程进入内核，0表示不限制。
并发值=CPU数量*磁盘数量*2
InnoDB使用两段处理来尝试让线程尽可能高效地进入内核。两段策略减少了因操作系统调度引起的上下文切换。线程第一次休眠innodb_thread_sleep_delay微秒，然后再重试。如果它依然不能进入内核，则放入一个等待线程队列，让操作系统来处理。
innodb_concurrency_tickets选项控制票据的数量。
innodb_commit_concurrency变量控制有多少个线程可以在同一时间提交。如
使用线程池(Thread Pool)来限制并发。

8.7 基于工作负载的配置

配置服务器的一个目标是把它定制得符合特定的工作负载。这需要精通所有类型的服务器活动的数量、类型，以及频率-不仅仅是查询语句，也包括其他的活动，例如连接服务器以及刷新表。
优化 BLOB和 TEXT的场景
服务器不能在内存临时表中存储BLOB值
通过SUBSTRING()函数把值转换为VARCHAR，或者让临时表更快一些。
优化排序(Filesorts)
MySQL有两种排序算法。如果查询中所有需要的列和ORDER BY的列总大小超过max_length_for_sort_data字节，则采用two-pass算法。或者当任何需要的列-即使没有被ORDER BY使用的列-是BLOB或者TEXT，也会采用这个算法。

第9章 操作系统和硬件优化

9.1 什么限制了MySQL的性能

9.2如何为MySQL选择CPU

可以通过检查CPU利用率来判断是否是CPU密集型的工作负载，但是仅看CPU整体的负载是不合理的，还需要看看CPU使用率和大多数重要的查询的I/O之间的平衡，并注意CPU负载是否分配均匀。
低延时(快速响应) + 高吞吐
多CPU在联机事务处理(OLTP)系统的场景中非常有用。
逻辑并发问题 + 内部并发问题
如果规划一个大的升级,必须要同时考虑硬件、服务器版本和工作负载。
归根结底，多个内核和多个物理CPU哪个更好，这是由硬件体系结构决定的。

9.3 平衡内存和磁盘资源

关键是要平衡内存和磁盘的大小、速度、成本和其他因素,以便为工作负载提供高性能的表现。
顺序IO比随机I/O快；存储引擎执行顺序读比随机读快。
多次写入，一次刷新；I/O合并
每个应用程序都有一个数据的“工作集”–就是做这个工作确实需要用到的数据。
工作集包括数据和索引，所以应该采用缓存单位来计数。一个缓存单位是存储引擎工作的数据最小单位。不同存储引擎的缓存单位大小是不一样的，因此也使得工作集的大小不一样。
一个高性能的I/O系统与带电池支持的写缓存或固态存储
快速访问时间 + 快速的传输速度 延迟 + 吞吐量
存储容量 传输速度 访问时间 主轴转速 物理尺寸
和CPU一样，MySQL如何扩展到多个磁盘上取决于存储引擎和工作负载。

9.4 固态存储

固态(闪存)存储器实际上是有30年历史的技术，但是它作为新一代驱动器而成为热门则是最近几年的事。
固态存储设备采用非易失性闪存芯片而不是磁性盘片组成。它们也被称为NVRAM，或非易失性随机存取存储器。
SSD(固态硬盘)和PCIe卡。
SSD通过实现SATA(串行高级技术附件)接口来模拟标准硬盘，所以可以替代硬盘驱动器，直接插入服务器机箱中的现有插槽。PCIe卡使用特殊的操作系统驱动程序，把存储设备作为一个块设备输出。
最重要的事情是提升随机I/O和并发性。
闪存的最重要的特征是可以迅速完成多次小单位读取，但是写入更有挑战性。
速度下降是由于没有空闲块时必须等待擦写完成所造成的。
两种主要的闪存设备类型:单层单元(SLC)和多层单元(MLC)。
平衡磨损和重映射的算法是非常重要的
对SATA SSD盘使用RAID(Redundant Array of Inexpensive Disks，磁盘冗余阵列)。
增加InnoDB的I/O容量； 让InnoDB日志文件更大；把一些文件从闪存转移到RAID；禁用预读；配置InnoDB刷新算法；禁用双写缓冲的可能；限制插入缓冲大小

9.5 为备库选择硬件

主备使用相同的硬件，或为主库购买新的硬件，然后让备库使用主库淘汰的老硬件。

9.6 RAID 性能优化

RAID不能消除甚至减少备份的需求。当出现问题的时候，恢复时间要看控制器、RAID等级、阵列大小、硬盘速度，以及重建阵列时是否需要保持服务器在线。
除了监控硬盘失效，还应该监控RAID控制器的电池备份单元以及写缓存策略。
操作系统、文件系统和操作系统看到的驱动器数量之间的相互作用可以是复杂的。Bug、限制或只是错误配置，都可能会把性能降低到远远低于理论值。
串行化可能发生在任何的软件或硬件堆栈层。
一是条带化阵列的块大小(Chunk Size)，还有就是控制器板载缓存(也称为RAID缓存)。
可以配置系统以便从应用到底层存储所有的块都对齐:InnoDB的块、文件系统的块、LVM，分区偏移、RAID条带、磁盘扇区。
RAID卡使用缓存:缓存读取；缓存预读数据；缓冲写入；内部操作

9.7 SAN 和 NAS

SAN(Storage Area Network)和NAS (Network-Attached Storage)是两个外部文件存储设备加载到服务器的方法。不同的是访问存储的方式。主要区别在于作为文件还是块设备访问存储
块接口在服务器上以逻辑单元号(LUN)或者虚拟卷(除非使用NFS)出现。许多SAN也允许多节点组成集群来获得更好的性能或者增加存储容量。
备份 简化容量规划 存储整合还是服务器整合 高可用 服务器之间的交互 成本

9.8 使用多磁盘卷

数据和索引文件 事务日志文件 二进制日志文件 常规日志(例如，错误日志、查询日志和慢查询日志) 临时文件和临时表
二进制日志和数据文件分离的真正的优势，是减少事故中同时丢失数据和日志文件的可能性。

9.9 网络配置

就像延迟和吞吐量是硬盘驱动器的限制因素一样，延迟和带宽(实际上和吞吐量是同一回事)也是网络连接的限制因素。
监控网络性能和所有网络端口的错误是正确的做法，要监控服务器、路由器和交换机的每个端口。

9.12 选择磁盘队列调度策略

在GNU / Linux上，队列调度决定了到块设备的请求实际上发送到底层设备的顺序。默认情况下使用cfq(Completely Fair Queueing，完全公平排队)策略。
查看系统所有支持的以及当前在用的调度策略:
$ cat /sys/block/sda/queue/scheduler

9.13 线程

MySQL每个连接使用一个线程，另外还有内部处理线程、特殊用途的线程，以及所有存储引擎创建的线程。
GNU/Linux提供两个线程库:LinuxThreads和新的原生POSIX线程库(NPTL)。

9.14 内存交换区

当操作系统因为没有足够的内存而将一些虚拟内存写到磁盘就会发生内存交换
极端的场景下，太多的内存交换可能导致操作系统交换空间溢出。
$ cat /proc/sys/vm/swappiness
修改存储引擎怎么读取和写入数据。
可以使用大页，不参与换入换出。
使用MySQ 的memlock配置项，可以把 MySQL锁定在内存。

9.15 操作系统状态

iostat和vmstat mpstat或者sar ifconfig netstat
默认情况下，vmstat和iostat只是生成一个报告，展示自系统启动以来很多计数器的平均值。两个工具都可以给出一个间隔参数，让它们生成增量值的报告，展示服务器正在做什么，这更有用。
procs(r b) memory(swpd free buff cache) swap(si so) io(bi bo) system(in cs) cpu(us sy id wa st)
rrqm/s和wrqm/s r/s和w/s rsec/s和wsec/s avgrq-sz avgqu-sz await svctm %util
利用利特尔法则(Little’s Law)得到下面的公式，计算出设备服务的并发请求数
concurrency = (r/s + w/s) * (svctm/1000)
dstat(http.:lldag.wieers.com/home-madeldstatl)）或collectl(http:l/collectl.sourceforge.net) blktrace pt-diskstats
密集型的机器
CPU密集型服务器的vmstat输出通常在us列会有一个很高的值，报告了花费在非内核代码上的CPU时钟﹔也可能在sv列有很高的值，表示系统CPU利用率
