第一章 MySQL架构与历史

不完美却足够灵活：Web类应用；嵌入到应用程序；支持数据仓库；内容索引和部署软件；高可用的冗余系统；在线事务处理系统(OLTP)
MySQL最重要、最与众不同的特性是它的存储引擎架构，这种架构的设计将查询处理(Query Prcocessing)及其他系统任务(Server Task)和数据的存储/提取相分离。

1.1 MySQL逻辑架构

MySQL服务器逻辑架构图
每个客户连接都会在服务器进程中拥有一个线程
认证基于用户名、原始主机信息和密码
优化与执行：解析查询，并创建内部数据结构(解析树)，然后对其进行各种优化，包括重写查询、决定表的读取顺序，以及选择合适的索引等.

1.2 并发控制

服务器层与存储引擎层
共享锁(shared lock 读锁 read lock)和排它锁(exclusive lock 写锁 write lock)
只会对修改的数据片进行精确的锁定
表锁 行级锁

1.3 事务

事务就是一组原子性的SQL查询，或者说是一个独立的工作单元
ACID：原子性(atomicity) 一致性(consistency) 隔离性(isolation) 持久性(durability)
四种隔离级别：READ UNCOMMITTED(未提交读) READ COMMITTED(提交读) REPEATABLE READ(可重复读) SERIALIZABLE(可串行化)
脏读可能性 不可重复读可能性 幻读可能性 加锁读
死锁是指两个或者多个事务在同一资源上相互占用，并请求锁定对方占用的资源，从而导致恶性循环的现象。
预写式事务日志(write-ahead logging)
MySQL提供了两种事务型的存储引擎：InnoDB和NDB Cluster
查看是否启用自动提交：SHOW VARIABLES LIKE “AUTOCOMMIT”
设置隔离级别：SET (SESSION) TRANSACTION ISOLATION Level
InnDB两阶段锁定协议(two-phase locking protocol)

1.4 多版本并发控制

MVCC 通过保存数据在某个时间点的快照来实现

1.5 MySQL的存储引擎

InnDB是MySQL的默认事务型引擎，也是最重要、使用最广泛的存储引擎。用来处理大量的短期(short-lived)事务
可测量性 可扩展性 可配置性 性能 各种新特性
InnDB表是基于聚簇索引建立的 InnDB的存储格式是平台独立的 可预测性预读 自适应哈希索引 插入缓冲区 支持真正的热备份
MyISAM: 全文索引 压缩 空间函数 不支持事务和行级锁 崩溃后无法安全恢复 数据文件(.MYD) 索引文件(.MYI) 可以包含动态或者静态行 加锁与并发 修复( check table table_name 检查错误 repair table table_name 修复) 索引特性 延迟更新索引键 myisampack对表进行压缩
Archive引擎 Blackhole引擎 CSV引擎 Federated引擎 Memory引擎 Merge引擎 NBD集群引擎
第三方存储引擎：OLTP类引擎 面向列的存储引擎 社区存储引擎
选择合适的引擎：事务 备份 崩溃恢复 特有的特性
日志型应用 只读或大部分情况下只读的表 电子公告牌和主题讨论论坛 CD-ROM应用 大数据量
转换表的引擎：ALTER TABLE; 导出与导入； 创建与查询
