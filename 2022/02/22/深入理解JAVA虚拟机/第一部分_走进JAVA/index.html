第一章 走进Java

概述
结构严谨、面向对象的编程语言；摆脱硬件平台的束缚，”一次编写，到处运行”；提供一种相对安全的内存管理和访问机制，避免绝大部分内存泄漏和指针越界问题；实现热点代码检测和运行时编译及优化；有一套完善的应用程序接口；第三方类库；
JAVA技术体系
JCP:Java Community Process “Java社区” 由业界多家技术巨头组成的社区组织，用于定义和发展Java的技术规范
JAVA技术体系：Java程序设计语言；各种硬件平台上的Java虚拟机实现；Class文件格式；Java类库API；来自商业机构和开源社区的第三方Java类库
JDK(Java Development Kit):支持Java程序开发的最小环境 = Java程序设计语言 + Java虚拟机 + Java类库
JRE(Java Runtime Environment):支持ava程序运行的标准环境 = Java SE API子集 + Java虚拟机
按照技术所服务的领域(或者按照技术关注的重点业务来划分)：Java Card; Java ME; Java SE; Java EE;
Java技术体系所包括的内容
JAVA发展史
Java语言的前身：Oak
JEP（JDK Enhancement Proposals）
虚拟机家族
虚拟机始祖：Sun Classic/Exact VM
Sun Classic：世界上第一款商业Java虚拟机
sunwjit(Sun Workshop JIT)-外挂编译器-SymantecJIT; shuJIT
Exact VM 准确式内存管理(Exact Memory Management)
武林盟主：HotSpot VM
Longview Technologies
热点代码探测技术
标准即时编译和栈上替换编译(On-Stack Replacement, OSR)
小家碧玉：Mobile/Embedded VM
CDC-HI(C Virtual Machine, CVM)和CLDC-HI(Monty VM)
CDC/CLDC(Connected (limited) Device Configuration)
HI(HotSpot Implementation)
Java ME的KVM
天下第二：BEA JRockit/IBM J9 VM
BEA JRockit:专门为服务器硬件和服务器应用场景高度优化的虚拟机；
IBM J9 VM：市场定位于HotSpot接近：服务端、桌面应用、嵌入式 IBM Technology for Java Virtual Machine
软硬合璧：BEA Liquid VM/Azul VM
与特定硬件平台绑定、软硬件配合工作的专有虚拟机，往往能够实现更高的执行性能，或提供某些特殊的功能特性
Liquid VM = JRockit VE(Virtual Edition, VE)
Azul VM在HotSpot基础上进行大量改进，运行与Azul Systems公司的专有硬件Vega系统上 PGC C4收集器
Zing Zulu
挑战者：Apache Harmony/Google Android Dalvik VM
Apache Harmony是一个Apache软件基金会旗下以Apache License协议开源的实际兼容于JDK 5和JDK 6的Java程序运行平台，未通过TCK(Technology Compatibility Kit)认证
如果一个公司要宣称自己的运行平台“兼容于Java技术体系”，那该运行平台就必须要通过TCK的兼容性测试，
Dalvik虚拟机曾经是Android平台的核心组成部分之一，并不是一个Java虚拟机，它没有遵循《Java虚拟机规范》，不能直接执行Java的 Class文件，使用寄存器架构而不是Java虚拟机中常见的栈架构。DEX(Dalvik Executable)
支持提前编译(Ahead of Time Compilation，AOT)的ART虚拟机
没有成功，但并非失败：Microsoft JVM及其他
微软为了在Internet Explorer 3浏览器中支持Java Applets应用而开发了 自己的Java虚拟机，只有Windows平台的版本
*百家争鸣 *
KVM：“Kilobyte” 简单、轻量、高度可移植，但是运行速度比较慢
Java Card VM：裁减了许多模块但通常支持绝大多数的常用加密算法。
Squawk VM：运行于Sun SPOT（Sun Small Programmable Object Tech-nology，一种手 持的Wi-Fi设备），也曾经运用于Java Card。
JavaInJava：试图以Java语言来实现Java语言本身的运行环境，既所谓的“元循环”(Meta-Circular，是指使用语言自身来实现其运行环境)虚拟机。
Maxine VM：一个几乎全部以Java代码实现(只有用于启动Java虚拟机的加载器使用C语言编写)的元循环Java虚拟机。从C1X编译器演进而来的Graal编译器
Jikes RVM：元循环虚拟机
IKVM.NET：基于微软.NET框架实现
展望Java技术的未来
无语言倾向
庞大的用户群和极其成熟的软件生态
Oracle Labs：Graal VM “Run Programs Faster Anywhere” “Universal VM”和“Polyglot VM”
在HotSpot虚拟机基础上增强而成的跨语言全栈虚拟机，可以作为“任何语言”的运行平台使用，
Graal VM可以无额外开销地混合使用这些编程语言，支持不同语言中混用对方的接口和对象，也能够支持这些语言使用已经编写好的本地库文件。
程序特化(Specialized，也常被称为Partial Evaluation):将这些语言的源代码(例如JavaScript)或源代码编译后的中间格式(例如LLVM字节码)通过解释器转换为能被Graal VM接受的中间表示(Intermediate Representation， IR)
Graal VM提供了Truffle工具集来快速构建面向一种新语言的解释器，并用它构建了一个称为Sulong的高性能LLVM字节码解释器。
新一代即时编译器
HotSpot虚拟机中含有两个即时编译器，分别是编译耗时短但输出代码优化程度较低的客户端编译器(简称为C1)以及编译耗时长但输出代码优化质量也更高的服务端编译器(简称为C2)，通常它们会在分层编译机制下与解释器互相配合来共同构成HotSpot虚拟机的执行子系统
Graal编译器
“Sea-of-Nodes”的高级中间表示(High IR)形式
“部分逃逸分析”(Partial Escape Analysis)
激进预测性优化(Aggressive Speculative Optimization)
向Native迈进
跨进程的、可以面向用户程序的类型信息共享(Application Class Data Sharing，AppCDS)，允许把加载解析后的类型信息缓存起来，从而提升下次启动速度
无操作的垃圾收集器(Epsilon，只做内存分配而不做回收的收集器，对于运行完就退出的应用十分合适)
逐步开始对提前编译(Ahead of Time Compilation，AOT)提供支持
Jaotc命令; Excelsior JET; Substrate VM;
灵活的胖子
HotSpot的定位是面向各种不同应用场景的全功能Java虚拟机
模块化是HotSpot的弱项，监控、执行、编译、内存管理等多个子系统的代码相互纠缠
ava虚拟机信息监控接口(Java Virtual Machine Profiler Interface，JVMPI)与Java虚拟机调试接口(Java Virtual Machine Debug Interface，JVMDI)
Java虚拟机工具接口(Java Virtual Machine Tool Interface，JVMTI)
Java语言级别的编译器接口(Java Virtual Machine Compiler Interface，JVMCI)
Java虚拟机的垃圾收集器接口(Java Virtual Machine Compiler Interface)
语言语法持续增强
一门语言的功能、语法又是影响语言生产力和效率的重要因素
实战：自己编译JDK
MacOS：XCode和Command Line Tools for XCode
Linux系统：GCC或CLang来进行编译
编译OpenJDK 12时， Bootstrap JDK必须使用JDK 11及之后的版本。
查询OpenJDK提供的编译参数：bash configure –help
configure命令承担了依赖项检查、参数配置和构建输出目录结构等多项职责
